## 1. 簡介

### 1.1 文件目的
本文件旨在詳細定義「AI 自動化新聞簡報系統」最小可行產品 (MVP) 的功能範圍、技術架構、用戶體驗以及開發標準。本文件將作為開發團隊（前端、後端、QA）與利益相關者之間的溝通基準，確保產品開發符合業務需求。

### 1.2 產品背景
在資訊爆炸的時代，獲取高品質且結構化的資訊變得日益困難。用戶每日需耗費大量時間瀏覽各大新聞網站、篩選感興趣的主題並手動整理摘要。為了提升資訊吸收效率並簡化「資訊獲取-內容轉化-知識分享」的工作流，我們需要一個自動化的工具來取代重複性的手動勞動。

### 1.3 產品願景
打造一個智慧化的個人情報中心，透過 AI 驅動的自動化流程，將海量新聞轉化為可立即使用的專業簡報，讓用戶從「找資訊」轉變為「決策與分享」。

### 1.4 MVP 目標
*   **自動化抓取**：每日定時從 Google News 抓取至少 8 個指定主題的新聞。
*   **精準摘要**：利用大語言模型 (LLM) 為每篇新聞產出具備高閱讀價值的內容摘要。
*   **結構化展示**：提供直觀的 Web 儀表板，按主題分類呈現當日資訊。
*   **自動化產出**：一鍵生成符合專業規格的 PPTX 簡報（一文一頁）。
*   **可視化檢閱**：在網頁端提供簡報與郵件內容的即時預覽。
*   **可控化發送**：實現「自動生成、人工確認」的郵件工作流，支援直接寄送與草稿同步。

## 2. 目標用戶

*   **主要用戶**：專業人士、產業分析師、投資者、熱愛特定主題（如 AI、體育）的知識愛好者。
*   **用戶特徵**：
    *   **時間敏感**：希望在 5-10 分鐘內掌握當日重點。
    *   **具備技術背景或專業知識**：對 AI 科技、財經動態有高度需求。
    *   **有分享需求**：需要將資訊整理後轉發給團隊或社群。
    *   **追求品質**：不滿足於單純的標題列表，需要深度摘要。

---

## 3. 核心問題與解決方案

*   **核心問題 1：資訊過載且分散**
    *   **MVP 解決方案**：透過自動化爬蟲整合 Google News 多主題資訊，並在統一的 Web 介面分類顯示，減少跨平台切換成本。
*   **核心問題 2：閱讀全文耗時過長**
    *   **MVP 解決方案**：導入 AI 摘要功能，將長篇文章濃縮為核心重點，讓用戶在短時間內判斷文章價值。
*   **核心問題 3：手動製作簡報報告極度費時**
    *   **MVP 解決方案**：開發自動化 PPTX 生成引擎，將新聞內容直接映射至簡報模板，實現「秒級」產出報告。
*   **核心問題 4：自動化寄件缺乏品質控管**
    *   **MVP 解決方案**：設計「人工確認」環節與「郵件預覽」功能，結合 Gmail/Outlook 草稿箱同步，確保寄出的資訊百分之百準確。

## 4. 產品功能 (MVP 範圍)

### 4.1 數據採集與處理模組 (Crawler & Processor)
*   **功能描述**：負責每日自動化獲取與過濾新聞數據。
*   **具體子功能**：
    *   **多主題關鍵字配置**：預設支持 AI 科技、科技產業、財經、股票、投資、理財、棒球、半導體、Web3、總體經濟等。
    *   **Google News 整合**：透過 RSS 或關鍵字搜尋 API 抓取各主題最新的 20 篇文章。
    *   **數據清洗**：過濾重複連結、無效標題及非相關內容。
    *   **自動排程任務**：每日定時啟動抓取任務（預設凌晨 04:00），確保用戶清晨可讀。

### 4.2 AI 內容摘要模組 (AI Engine)
*   **功能描述**：利用 LLM 對原始文章進行深度加工。
*   **具體子功能**：
    *   **內文抓取**：自動解析新聞連結內容。
    *   **多維度摘要生成**：生成包含 3-5 個要點的繁體中文摘要。
    *   **元數據提取**：自動提取發布日期、來源媒體名稱及原始連結。
    *   **異常處理**：若 AI 處理失敗，系統應記錄錯誤並保留原始標題供用戶手動處理。

### 4.3 Web 管理儀表板 (Dashboard)
*   **功能描述**：用戶瀏覽與操作的核心介面。
*   **具體子功能**：
    *   **主題分類標籤頁 (Tabs)**：以不同頁籤區分 AI、財經、棒球等主題。
    *   **新聞卡片清單**：顯示標題、摘要、來源及原始連結按鈕。
    *   **狀態標記**：標記哪些文章已包含在簡報中，哪些尚未處理。
    *   **手動刷新**：提供手動觸發重新抓取與摘要的按鈕。

### 4.4 簡報生成引擎 (PPTX Generator)
*   **功能描述**：將結構化數據轉換為 PowerPoint 文件。
*   **具體子功能**：
    *   **一文一頁邏輯**：每一篇新聞自動生成一張 Slide，包含標題、摘要、來源及 QR Code/連結。
    *   **簡報模板套用**：使用預設的簡潔商務風格模板。
    *   **網頁端簡報預覽**：整合 Canvas 或 PDF 預覽組件，讓用戶在網頁上翻閱生成的 PPTX。
    *   **下載功能**：支持用戶直接下載產出的 PPTX 檔案至本地。

### 4.5 郵件工作流 (Email Workflow)
*   **功能描述**：負責內容的分發與分享。
*   **具體子功能**：
    *   **模式切換**：提供「API 直接寄送」與「同步至草稿匣」兩種模式切換。
    *   **郵件內容預覽**：點擊寄出前彈窗顯示郵件內文、收件人列表及附件預覽。
    *   **收件人管理**：簡單的通訊錄功能，記錄常用寄送名單。
    *   **人工確認機制**：所有自動產生的郵件必須點擊「確認寄出」後方能發送。

## 5. 用戶故事

1.  **作為** 一名科技分析師，**我想要** 每天早上看到 AI 自動抓取的半導體新聞，**以便** 我能快速掌握產業脈動而不需翻閱多個網站。
2.  **作為** 一名理財顧問，**我想要** 系統為每篇財經新聞生成繁體中文摘要，**以便** 我能快速篩選出值得推薦給客戶的資訊。
3.  **作為** 一名棒球愛好者，**我想要** 系統將昨晚的比賽報導整理成簡報，**以便** 我能在通勤時快速翻閱重點。
4.  **作為** 一名團隊主管，**我想要** 在網頁上直接預覽產出的 PPTX，**以便** 我確認內容無誤後再進行分發。
5.  **作為** 一名內容管理員，**我想要** 在寄出信件前先預覽信件內容，**以便** 確保附件與內文格式正確。
6.  **作為** 一名謹慎的用戶，**我想要** 系統將郵件先存入我的 Gmail 草稿匣，**以便** 我能做最後的文字微調再寄出。
7.  **作為** 一名開發者，**我想要** 系統能自動過濾重複的新聞，**以便** 保持簡報內容的精煉與高品質。

## 6. 非 MVP 範圍 (未來考量)

1.  **自定義數據源**：允許用戶自行輸入特定的 RSS URL 或特定網站（如 TechCrunch）。
2.  **多國語言支援**：支援將外文新聞（英/日）自動翻譯並摘要為中文。
3.  **PPTX 模板自定義**：用戶可上傳自己的品牌 PPTX 模板進行動態填充。
4.  **手機 App 應用程式**：開發 iOS/Android 版本以支援行動端推送。
5.  **付費訂閱系統**：針對進階主題或大量抓取需求實行 SaaS 訂閱機制。
6.  **深度數據分析**：利用 AI 分析各主題的熱度趨勢圖表並插入簡報。
7.  **社交媒體自動分享**：一鍵分享摘要至 LinkedIn、Twitter 或 LINE 群組。
8.  **多格式導出**：支援導出為 PDF、Notion 頁面或 Markdown 文件。
9.  **語音摘要播報**：將摘要轉化為語音 (TTS)，供用戶在開車時聽取新聞。
10. **智能去重與聚類**：將相似主題的多篇報導自動合併為一個專題頁面。

## 7. 技術要求

### 7.1 後端框架
*   **建議**：Node.js (NestJS 框架)
*   **理由**：NestJS 提供強大的模組化結構，適合處理爬蟲任務與 API 整合；與前端 React 共享 TypeScript 定義可提高開發效率。

### 7.2 資料庫
*   **建議**：PostgreSQL
*   **理由**：穩定且支援 JSONB 格式，適合存儲結構化的新聞數據與用戶配置。

### 7.3 資料庫 ORM
*   **建議**：Prisma
*   **理由**：強型別支援，能大幅減少資料庫操作的錯誤，且具備優秀的遷移工具。

### 7.4 前端技術
*   **建議**：React + Tailwind CSS + Vite
*   **理由**：React 生態系豐富，適合開發複雜的儀表板與預覽組件；Tailwind CSS 可加速 UI 開發。

### 7.5 其他整合
*   **爬蟲引擎**：Puppeteer 或 Playwright (用於處理動態網頁內容)。
*   **AI 接口**：OpenAI API (GPT-4o 或 GPT-4-turbo)。
*   **簡報庫**：`pptxgenjs` (用於後端或前端生成 PPTX)。
*   **郵件服務**：Nodemailer (用於 SMTP) 或 Gmail/Outlook API。

### 7.6 部署環境
*   **建議**：Docker 容器化部署於 AWS (EC2/RDS) 或 Vercel (前端) + Render (後端)。

### 7.7 版本控制
*   **建議**：Git (GitHub)，並採用 Git Flow 工作流。

## 8. 成功指標

1.  **效率提升**：用戶從獲取資訊到產出 PPTX 簡報的時間由平均 60 分鐘降低至 5 分鐘內。
2.  **爬取達成率**：每日成功抓取並處理的新聞比例需達 95% 以上。
3.  **摘要準確率**：透過人工抽樣檢查，AI 摘要能準確表達原文核心價值的比例需大於 90%。
4.  **用戶滿意度**：MVP 測試期用戶對於簡報排版與摘要質量的評分平均需達 4.0/5.0。
5.  **功能完成度**：100% 實現本文件定義的所有 MVP 核心功能模組。

## 9. 假設與約束

### 9.1 假設
1.  **Google News 可存取性**：假設 Google News 的 RSS 或搜尋結果介面不會發生大規模阻擋或格式突變。
2.  **AI API 穩定性**：假設 OpenAI API 保持穩定且延遲在可接受範圍內（摘要生成 < 10 秒）。
3.  **內容授權**：假設抓取摘要與標題並註明來源符合合理使用原則。
4.  **用戶環境**：假設用戶使用的是現代網頁瀏覽器（Chrome, Edge, Safari）。
5.  **網路環境**：伺服器端具備穩定的網路連接以執行大规模爬蟲。

### 9.2 約束
1.  **法律限制**：必須遵守各新聞網站的 `robots.txt` 規範，避免過度抓取導致 IP 被封禁。
2.  **API 成本**：每月 AI API 的使用額度需控制在預算範圍內。
3.  **技術限制**：由於 PPTX 檔案格式複雜，網頁端預覽可能與實際下載的檔案有些微樣式偏差。
4.  **開發週期**：MVP 開發週期限制在 4-6 週內完成。
